# 訊息面步驟追蹤（step-trace）

每一則對使用者的訊息都適用的無序規定。目的：讓使用者從任何一則訊息都能還原「現在跑到 SOP 第幾步、中間跳過了什麼」。

## Rule 1 — 首行恆為 Step 標記行

- 每則訊息第一行固定為 Step 標記行，格式：`Step X–Y 完成；Step Z 進行中`（無累積區間時 `Step Z 進行中` 即可；停點加「停點」字樣）。
- 其他一切內容——開場白、對外部輸入的回應、子技能的措辭框架——一律放在標記行之後。

### Good

情境: 收到使用者拍板「A」後的第一則訊息。

```
Step 6.2 完成（拍板 A）；Step 6.2 處置進行中
收到拍板：本輪緩做。依結論我會把澄清紀錄寫入 spec.md……
```

### Bad

情境: 同上。

```
收到拍板：A——本輪緩做。依 Step 6.2 處置：……
```

結果: 標記被「收到拍板」前綴擠到句中，使用者無法從首行定位進度。

預期改法: 把「收到拍板」等回應語放到 Step 標記行之後。

## Rule 2 — 回應外部輸入的訊息不豁免

- 「收到指示」「收到拍板」「收到稽核回報」這類回應外部輸入的第一則訊息，最容易漏標記——它們同樣適用 Rule 1，首行先標記、再回應。

### Good

```
Step 8 完成（稽核零 violations）；Step 9 進行中
收到稽核回報：通過。本輪無 entry 落地，直接進入 matrix 回寫……
```

### Bad

```
收到稽核回報：……Step 8 完成，續跑 Step 9–10。
```

## Rule 3 — 短過場訊息也要標記

- 「執行 worklist 查詢」「重新讀取後再編輯」這類一兩句的過場訊息，若無標記會造成編號斷點；短訊息同樣以標記行開頭，或併入下一則訊息不單獨輸出。

### Good

```
Step 3 進行中
CLI 用法已讀，執行 worklist 查詢。
```

### Bad

```
CLI 用法已讀。執行 worklist 查詢。
```

## Rule 4 — 連續 tool 步驟的區間補宣告

- 多個 step 只有 tool 呼叫、無中間訊息時，下一則訊息的標記行必須以區間一次補宣告（例：`Step 4–7 完成；Step 8 進行中`），不得漏報任何步；不得等到結尾才總結補報。

## Rule 5 — 停點與結尾報告必含當前步

- 停點訊息（問 plan package、clarify 批次）與結尾報告的標記行必含當前步編號（例：`Step 2 停點`、`Step 11 結尾報告（Step 1–10 已全數完成）`）。
