# Atomic Rule 顆粒度

- 一條 rule 永遠掛在一個 Action 上：每條 rule 都從上游 `/aibdd-flows-specify`（萃取 api-wise 業務 action 時綁定）之 Action 出發；脫離 Action 漂浮的 rule 不得收下，一條 rule 跨多個 Action 的也不得收下。
- 一條 rule 等於一個類型前綴下的一個原子斷言：rule 名稱必須命中下節「4 種類型前綴」之一，且只承載單一主詞、單一動詞、單一條件；複合斷言一律拆分（判定見「原子化判定」）。
- 每個 Feature 至少要有一條「前置（參數）」rule：用以驗證必要參數；所有「缺少必要參數」之檢查得合併為單一前置（參數）rule（下一階段以 Scenario Outline 展開），其他領域特定之參數約束仍各自獨立為原子 rule。
- 必須能指回需求原句：每條 rule 之主詞、條件都要在需求敘述或其對應 impact 的 quotes 中找得到對應出處；不得憑空長出原文沒有的內容（自生的判定見下節「禁止自生」）。
- 需求列舉須逐列展開：需求上下文內的表格、分級、允許值、回應欄位等列舉，須逐列展開成獨立 rule 或掛在 rule 下的補充行，不得概括成單條高階 rule。
- 需求明示的前置與結果須顯式收錄：quote 明示的硬前提（如須先登入、某狀態須成立）與可驗收結果，須逐一收成對應類型前綴的 rule，不得只覆蓋一部分或隱含帶過（例：需求寫「付款成功時要建立訂單並發送確認通知」，不得只收 `訂單應建立` 而漏 `應發送確認通知`）。
- 語意不得偏離原句：rule 的主詞、條件與結果不得與需求原句相反、縮小或放大（語氣強度的判定見下節「禁止自生」之 hedging 條款）。
- 主詞責任歸屬須明確：rule 主詞須可判定為 actor、aggregate、system 或 external dependency 之一，誰負責記錄、驗證、轉移狀態或對外回應不得混淆（例：「應記住上一次配送選項」須明確是購物車、顧客帳戶還是 UI 暫存記住）。
- rules 彼此不得矛盾：同一 `.feature` 內的 rules 不得互相否定，或同時要求不可並存的結果；表面衝突通常是缺了前提、適用範圍或例外情境，須釐清解除而非並存（例：一條寫 `購物車商品數量必須大於 0`、另一條寫 `購物車必須為空`）。
- 疊加須有合成規則：多條 rule 同時作用於同一計算、狀態轉移或決策點時，須有一條更高層 rule 定義疊加順序、優先權或合成方式（例：折價券與折扣券同時使用時，先折價再折扣還是先折扣再折價）。
- 掛在單一 Feature 檔上：每條 rule 對應的目標 `.feature` 是 function package 之下的單一檔；不得指向 boundary 根的 `features/`，也不得同時掛多個檔。
- 同一 `.feature` 只承載一個 operation：當同一個 Feature 檔被多個可獨立驗收、獨立命名、獨立觸發的 operation 共用時，必須標出來進澄清，不得直接交棒。
- 不夾帶資料、Scenario、Background：rule 只描述在什麼前置下、必須或應符合什麼斷言；範例資料、Background、Scenario、Examples、表格展開都留給下一階段（Scenario formulation）。
- 去重：同一個 `.feature` 上、同類型前綴加同主詞加同條件者收成一筆；兩條 rule 僅為換句話說、或一窄一寬而無獨立驗收價值者亦屬重複，保留語意最完整、最直接對應需求原句的一條。
- command 與 query 必須分檔：有寫入或副作用之規則與純查詢之規則不得綁在同一 `.feature`；違者拆檔。
- 保留 `@ignore`：上游 `.feature` 帶 `@ignore` 標記時，對應 rule 仍寫進草稿（保留標記），下一階段才決定是否實作。

## Rule 的 4 種類型前綴

每條 rule 名稱必須遵循下列命名句型，類型前綴只能命中以下 4 種之一；前置用「必須」（約束），後置用「應」（預期結果）：

```
Rule: <類型前綴> - <主詞> 必須／應 <單一條件>
```

| 類型前綴 | 語意 | 動詞 | 範例 |
|---|---|---|---|
| 前置（狀態） | 觸發前系統或聚合必須處於什麼狀態 | 必須 | `Rule: 前置（狀態）- 商品庫存必須大於 0` |
| 前置（參數） | 觸發時輸入參數必須符合什麼約束 | 必須 | `Rule: 前置（參數）- 加入數量必須大於 0` |
| 後置（回應） | 觸發完成後對外回應應為什麼 | 應 | `Rule: 後置（回應）- 操作失敗應回傳「庫存不足」` |
| 後置（狀態） | 觸發完成後系統或聚合狀態應變成什麼 | 應 | `Rule: 後置（狀態）- 購物車應新增一筆商品項目` |

## 原子化判定

- 陳述句中出現「且」「和」「並且」則拆分為多條 rule。
- 陳述句中出現「或」則保留在同一條 rule。
- 同一條 rule 同時混合前置與後置則拆分為前置一條、後置一條。
- 一條 rule 出現兩個主詞則拆分為各自一條。

# 反例

- `Rule: 前置（狀態）- 商品庫存必須大於 0 且商品狀態必須為上架中`：兩個主詞、兩個條件，違反原子化，應拆為兩條 `商品庫存必須大於 0`、`商品狀態必須為上架中`。
- 一條 rule 寫成「使用者送出訂單時，系統會驗證金額、扣款、寄信、寫入訂單表」：多前置與多後置混在一句，應拆成 前置（狀態）- 購物車金額必須大於 0、後置（狀態）- 訂單應建立、後置（狀態）- 帳戶餘額應扣除訂單金額、後置（回應）- 系統應回傳訂單成立通知，各自掛回對應 Action。
- 一條 rule 寫成「系統應正確處理付款」：主詞、條件都模糊，類型前綴也命不中任何一種，不收。
- rule 寫進「例如：金額 等於 100、餘額 等於 50 則拒絕」：範例資料屬於下一階段 Scenario，不在此處出現。
- 同一個 `.feature` 同時掛 建立訂單 與 取消訂單 兩個可獨立驗收或觸發的 operation 之 rule：應標為多 operation 混檔進澄清，不直接交棒。
- 一條 rule 寫成「商品庫存應大於 0」：前置用「應」、語氣弱化成預期，違反前置用必須、後置用應的動詞約定。
- 一條寫入訂單的 下單 rule 被綁在僅描述 查詢訂單列表 之 `.feature`：command 綁到 query 檔，違反 command 與 query 分檔。
- 一個 Feature 完全沒有「前置（參數）」rule：違反每個 Feature 至少要有一條前置（參數）rule。

## 禁止自生（quote 沒授權、rule 不得自己長出來）

每條 rule 中下列四類內容，只要在需求原句或其對應 impact 的 quotes 中找不到對應出處，就視為 rule 自生、不得收下：

1. 動詞與介面詞：quote 用業務動作詞時，rule 不得自己長出 HTTP method、API verb、status code、路徑樣式。
   - 例：quote 寫「建立會員資料」，rule 不得寫成「`POST /members` 回傳 201」。
2. 數值與技術欄位：quote 沒提到的數字或欄位（SLA、retry、timeout、版本號、過期時間、數值上下限、通道如 email 或 SMS 等），rule 不得自填。
   - 例：quote 寫「失敗會通知使用者」，rule 不得寫成「失敗時於 30 秒內以 email 通知，最多重試 3 次」。
   - 例：quote 寫「金額不可為負」，rule 不得補上「金額介於 1..100,000」。
3. 隱性前提：quote 沒寫的前提條件（先登入、先 KYC、先綁定信用卡、先同意條款等）不得被 rule 自己加成阻擋條件。
   - 例：quote 寫「會員可下單」，rule 不得寫成「已通過 KYC 之會員可下單」。
4. 與原句衝突的語氣（hedging）：quote 已明確分類（必須、不得、一律、僅）時，rule 不得弱化成「通常」「視情況」「可能」「大多會」等模糊語氣；反之亦然。
   - 例：quote 寫「必須通過二階段驗證」，rule 不得寫成「通常會通過二階段驗證」。
