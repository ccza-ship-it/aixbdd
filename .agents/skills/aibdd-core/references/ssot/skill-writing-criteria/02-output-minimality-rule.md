# Rule 1 - 移除動作不留殘影

- Description:
  1. 移除某動作後，產出中不得殘留任何指向它的禁令（「不再做 X」「此步不再…」）；該動作要在最終檔裡完全消失，而非留一句殘影。
- Rationale:
  1. 殘影逼讀者反推被刪的是什麼，等於要靠時光機回推被移除的內容；直接刪除才讓檔案維持 as-is。

## Good

1. 某 SKILL.md 移除 legacy-path 寫入後，整份檔搜不到任何「legacy path」字樣（動作乾淨消失，無殘影）。
2. 移除「WRITE 到 legacy path」這個動作時，直接刪掉那一整步、其餘步驟重新編號，產出只剩乾淨步驟序列。

## Bad

1. 移除動作後，下一步括號留著「（此步不再寫 legacy path）」（殘影禁令；產出該讓被刪動作完全消失，否則讀者得反推被刪的是什麼）
2. 步驟主體刪掉了，卻在檔尾 changelog 式地補一句「移除了 legacy path 寫入」（殘影；source artifact 不留決策脈絡，被刪的就該無痕）

# Rule 2 - 立法集中（不重複、不散落）

- Description:
  1. 同一條禁令／規則不得在產出的多處重複出現（例如 PRINCIPLE 一份、各步驟括號又各抄一份）；只看最終檔，重複即違規。
  2. 跨步驟的 invariant 要集中在單一 `## PRINCIPLE`，不得散落寄生在各步驟的括號提示裡。
- Rationale:
  1. 重複立法在某處漏改時會留下多份不同步的副本互相矛盾，讀者無法判斷哪份才算數。
  2. invariant 集中宣告一次才有單一可回指的位置；散在括號裡等於同一條法律要讀完所有步驟才拼得出全貌。

## Good

1. 「不得 raw-read yaml」只出現在 `## PRINCIPLE: DSL 只透過 dsl_cli query 取得` 一處，各步驟不再各自重述。
2. 跨步驟的「所有路徑用 `${}` placeholder」寫成一條 `## PRINCIPLE`，各步驟只管自己的動作。

## Bad

1. 「不得 raw-read yaml」同時出現在 PRINCIPLE 與三個步驟的括號（重複立法；某處漏改時四份副本互相矛盾）
2. 跨步驟 invariant「所有路徑用 `${}` placeholder」被拆塞進五個步驟的括號提示（散落寄生；應集中成一條 PRINCIPLE，否則讀者要掃完五步才知道有這條法律）

# Rule 3 - 無雜訊行

- Description:
  1. 產出中某行若刪掉後 skill 的合法 action set 與行為不變，那行就不該存在（例如不影響任何下游的補充括號註）。
- Rationale:
  1. skill 會被執行非常多次，每一行沒在拉動價值的內容都是永久成本。

## Good

1. 每個步驟的括號只放該步自身、會影響行為的參數說明，沒有純補充性的旁註。

## Bad

1. `READ arguments.yml（注意它在 repo 根）` 這行括號註不影響任何下游行為（雜訊行；刪掉後 action set 與行為不變）
2. 步驟之間插入一行「以下開始進入驗證階段」之類的旁白（雜訊行；不構成任何動作，刪掉行為不變）
